# B2B 智能补货系统 V2.0 架构蓝图 (中央总盘预言机)

> **版本主旨**：从“极度稀疏的微观节点预测 (V1.x)” 全面跃升为“密集的宏观商品池预测 (V2.0)”。直接对齐商品部“单款货品未来 30 天要不要追单、追多少”的终极痛点。

---

## 1. 业务愿景与目标的变迁 (V1.x vs V2.0)

### 🚨 V1.x 的阶段性历史使命
- **底层逻辑**：`(买手ID, SKU_ID) × 历史买手行为` ➜ `单个买手30天内在该款的补货量`。
- **成就**：跑通了整个数据流，证明了可以用 AI 从数十万小单中沙里淘金，给出定量依据。
- **痛点**：数据极度稀疏（99% 的 30天时间窗内，单个买手不会补货同一件衣服），容易受单店偶然性干扰。商品部看最终大盘量，还要加总一遍。

### 🚀 V2.0 的直接击杀 (Direct Hit)
- **底层逻辑**：`SKU_ID × 全国买手聚合大盘行为` ➜ `全网30天内对该款的总补货量`。
- **价值**：不再去管“哪个具体买手”会买，而是死死盯住**“这款衣服在全国的热度大盘”**。输出直接就是：`SKU: AM31001553，预测未来 30 天全国将发生 540 件现货补货，建议备库 600 件`。

---

## 2. 数据降维与高质化 (Data Distillation)

V2.0 最大的红利在于**数据量骤减，但数据质量百倍提升**。

*   **V1.x 原本的数据张量**：1,000 个买手 × 250 个 SKU × 365 天 = 极其稀疏，每天矩阵大量的 0。
*   **V2.0 新的数据宽表**：250 个 SKU × 365 天 = 非常紧凑致密。每天每一行都是该款衣服在全国各处的真实跳动。

### 重新定义 V2.0 黄金特征集 (The Golden Features)

我们把之前商品部**人工判断补货**的核心经验，直接沉淀为 V2.0 的底层特供变量：

**A. 核心人工经验向量化 (Manager's Intuition)**
*   **铺店率 (Coverage_Rate)**：当前历史时间锚点前，全国提过这款货的买手占总买手的比例。
*   **店均发货量 (Avg_Shipped_per_Store)**：全国总发货量 / 提过货的买手数。
*   **店均补货量 (Avg_Replenish_per_Store)**：全国总补货量 / 发生过补货的买手数。
*   **补货率 (Replenish_Ratio)**：发生过补货的买手 / 提过该款的买手总数。

**B. 流量环境指标 (Macro Traffic Signals)**
*   **大盘加购度 (Cart_Add_7d/30d)**：该款过去 7/30 天在小程序被全国买手点击、加入购物车的总计次数（极强的前置先兆指标）。

**C. 商品深度 DNA (Item DNA)**
*   **首发面世天数 (Age_of_SKU)**：计算锚点日距离该 SKU 首次在系统产生单据的天数。（解决核心生命周期认知：刚发售 3 天的新款，与已经发售 3 年的老款，同样的斜率，后续走势天差地别）。
*   **自然属性**：品类、季节、价格带。

## 3. 模型选型与架构重构 (Model Evolution)

## 3. 模型选型：保留沿用 LSTM 双塔，实施「降维打击」

用户的核心诉求是**尽量不动深层的 LSTM 模型代码，而是通过改动输入端的数据流来完成 V2.0 的跃迁**。这是一条工程成本极低、性价比极高的路径。

### 方案 A：LSTM 架构复用 + 数据流上游聚合 (高性价比路线)

**核心原理：**
目前的模型输入是 `[买手ID, SKU_ID, 90天时序点, 7维特征]`。
V2.0 保持这个输入结构完全不变。我们将所有的 `买手ID` 强行变成一个伪造的超级常量（比如 `buyer_id = 99999` 代表“全国大盘”）。
在 ETL 阶段（`build_wide_table.py`），我们在时间流上将所有属于同一 `SKU_ID` 的发货量进行 `SUM` 聚合。

- **优势 1 (零代码推翻)**：不需要重写复杂的 `generate_daily_inference.py` 推断逻辑，不需要更换损失函数，也不用重整 DataLoader，原有的模型工程管线 100% 复用。
- **优势 2 (填平零膨胀)**：原本某买手经常断断续续发生 0 销量，被聚合后，这个全国虚拟大盘买手的序列将变成非常致密、连续的波浪线，完美匹配 LSTM 抓取周期惯性的能力。
- **优势 3 (解锁上限)**：模型学习不再被截断。我们只需修改 `evaluate_dashboard.py` 中的 `preds = torch.clamp(preds, max=8.5)`，把它拉长到 `max=11` (意味着支持预测1万件以上的单日发货量)，这就足以拦截大爆单。

### 方案 B：时序卷积 / 轻量化 Transformer 编码器 (远期演进)
如果在跑了全国聚合版 LSTM 之后，发现仍然对短期极速突变（如某天直播带货导致销量的垂直飙升）捕捉不够敏感，再去考虑引入一维 CNN 或引入外部全局特征（如气象、大促日标记）补充作为第三个信息塔。

## 4. 新品冷启动终极解法 (Zero-Shot for Cold Start)

在 V2.0 提升到 SKU 全图视角后，纯新品预测终于有了物理意义上的解法（不再是无脑给 0）。

**破冰原理：款式 KNN（K-Nearest Neighbors）与“首单初动关联”**
1.  **特征寻亲**：当季上架一个毫无历史经验的新款（比如一款 S 线夏装蕾丝短裙）。
2.  **聚类图谱**：系统在往年历史库中，根据其`价格带`、`品类`、`尺码特征`、`上市月份`寻找最接近的历史 5 个老款。
3.  **曲线嫁接**：用老款曾经的“首置前 30 天生命周期补货曲线”，结合新款在期货订货会上的表现，加权生成冷启动备货预估。
4.  或者直接在系统中，专门为 `Age_of_SKU <= 7` 的衣服单独跑一个“纯冷启动起量预估网络”。

## 🚀 结论与路线演进建议

1.  **立即稳住盘面**：用 V1.8 成果作为 B2B 补货系统的第一座里程碑完成跨部门交差，确立 AI 能在这一复杂领域从无到有建立数据管线并跑通预测基线。
2.  **启动清洗转向**：不用等代码架构大改，**只改 `clean_data.py` 和 `build_wide_table.py` 的 GroupBy 维度**（从买手级 GroupBy 变为 SKU 级聚合），就能快速得到致密的 V2.0 大盘宽表。
3.  **轻骑兵验证**：用 Python 调包 sklearn/LightGBM 对新的 SKU 大宽表进行速跑验证，一周内就能看到 Ratio 和 RMSE 是否产生指数级飞跃。
