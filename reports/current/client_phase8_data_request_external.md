# B2B 补货预测项目：Phase8 数据确认与补充请求

## 1. 本轮沟通目的

当前项目的模型主线已经稳定，下一阶段不会重新更换模型路线，而是针对剩余误差做定向优化。  
这轮需要甲方协助的重点，不是再补大量交易流水，而是：

1. 确认当前关键原始表的业务语义
2. 补充少量高价值表或字段
3. 让后续优化建立在更准确的数据定义上

本轮最主要想解决的模型剩余问题是：

- 高需求大货仍有少量漏补
- 弱信号但真实需求很高的 SKU 仍然最难判断
- 部分类目在特定月份的大货场景下仍偏保守
- 少量本来不需要补货的 SKU 仍会被预测出小额补货

## 3. 本轮优先级最高的事项

### 3.1 订单表语义确认

涉及原始表：

- `V_IRS_ORDERFTP.csv`

当前我们已经在使用这张表做订单建模，但还有 3 个地方需要先确认：

#### A. `TYPE` 缺失的含义

需要确认：

1. `TYPE` 为空时，是否有固定业务含义
2. 这部分是否默认属于某一类订单，例如“现货”或“期货”
3. 是否有其他字段可以帮助补齐 `TYPE`

#### B. `QTY < 0` 的含义

需要确认：

1. 负数数量是否代表撤单、冲销、改单、退货、调拨回退或其他动作
2. 负数记录是否通常对应一笔原始正数订单
3. 是否存在可关联的原单号、关联单号、变更原因、操作类型

#### C. 重复行的含义

需要确认：

1. 完全重复行是否代表真实多次业务动作
2. 还是系统导出重复
3. 是否存在唯一单据号、行号、流水号可以用于判重

这部分很重要，因为它直接影响模型训练标签是否准确。

---

### 3.2 生命周期表

这张表是本轮最希望新增的一张结构化表。

#### 这张表的作用

主要用于区分：

- 新品刚启动，应该放量
- 稳定款，正常补货
- 尾货/临近下架，应该保守

这对“历史信号弱，但未来真实补货量很高”的款式判断非常关键。

#### 建议优先按 `style_id` 提供

如果生命周期主要按款级管理，建议以 `style_id` 为主。  
如果某些 SKU 在生命周期上与同款其他 SKU 差异明显，再补 `sku_id`。

#### 建议字段

| 字段 | 含义 | 必需性 |
| --- | --- | --- |
| `style_id` | 款号主键 | 必需 |
| `sku_id` | 条码级 SKU 主键 | 可选 |
| `launch_date` | 正式上市日期 | 必需 |
| `first_orderable_date` | 买手第一次可下单日期 | 推荐 |
| `first_delivery_date` | 第一次发货日期 | 推荐 |
| `planned_off_shelf_date` | 计划下架日期 | 必需 |
| `actual_off_shelf_date` | 实际下架日期 | 推荐 |
| `is_carry_over` | 是否延续款 | 推荐 |
| `is_reorderable` | 是否允许返单/追单 | 必需 |
| `product_stage` | 新品 / 起量期 / 稳定期 / 清仓期 | 推荐 |
| `is_key_style` | 是否主推款 | 推荐 |
| `season_type` | 春夏 / 秋冬 / 四季等 | 推荐 |

#### 最低可接受版本

如果暂时无法提供完整生命周期表，最低也希望提供：

- `style_id`
- `launch_date`
- `planned_off_shelf_date`
- `is_reorderable`

### 3.3 预售表字段语义确认

涉及标准表：

- `V_IRS_PREORDER`

根据甲方提供的表结构说明，目前这张表里最关键的字段定义已经基本明确：

- `name` = 商品 / 款号 `skc`
- `no` = 条码 / `sku`
- `customername` = 客户名称
- `storename` = 客户店仓名称
- `qty_rem` = **待转现货数量**

这里的 `qty_rem`，不建议再理解成模糊“剩余量”，而应理解为：

- **后续会转成现货需求的待转数量**

这部分对后续模型优化价值很高，因为它比普通浏览、加购更接近真实补货意图。

当前还需要确认的是：

| 字段 | 需要确认的问题 |
| --- | --- |
| `customername` / `storename` | 分别代表客户主体还是实际门店/买手 |
| 增量更新时间字段 | 是否存在，例如 `modifieddate` / `creationdate` |
| 状态字段 | 是否存在，用于判断是否仍为“正常、待转现货” |

如果还有更完整版本，也希望补充：

- `presale_qty`
- `presale_order_cnt`
- `presale_buyer_cnt`
- `presale_create_time`
- `presale_status`

## 4. 已有数据，希望补充字段或语义

### 4.1 库存每日快照

这部分我们已经保留了 2026 年开始的每日库存快照。  
根据甲方表结构文档，当前库存建议区分为两类：

#### A. 现货可用库存

标准表：

- `V_IRS_STORAGE`

关键字段：

- `name`
- `no`
- `qty_can`

文档口径明确为：

- `qty_can = 现货可用库存`
- 且库存店仓取：
  - `0034-AVVENN买手备量仓`
  - `0004-REVAN买手备量仓`

这类库存优先级最高，因为它直接对应买手现货补货可配量。

当前我们已经确认有对应快照文件，例如：

- `snapshot_inventory/20260130_storage_stock.csv`

#### B. B2B 预售发布库存

标准表：

- `V_IRS_B2BSTORAGE`

关键字段：

- `NAME`
- `NO`
- `QTY_HQ`

这部分更接近：

- 总部发布库存 / 预售发布库存

它和现货可用库存不是一个概念，建议后续分开使用。

当前我们已经确认有对应快照文件，例如：

- `snapshot_inventory/20260130_b2b_stock.csv`

当前库存部分，我们倾向于继续沿用现有快照口径，不建议在项目中途修改已经在跑的库存表结构或更新方式。  
本轮更希望确认的是：

1. `storage_stock` 是否正式对应 `V_IRS_STORAGE`
2. `qty_can / qtycan` 是否稳定表示“现货可用库存”
3. `b2b_stock` 是否正式对应 `V_IRS_B2BSTORAGE`
4. `qty_hq` 是否稳定表示“总部发布库存 / 预售发布库存”
5. 这两类快照后续是否会继续按同样结构日更

如果后续还有更完整库存字段，建议作为新增表或新增版本单独补充，而不是直接改当前表结构。

如果甲方系统中未来还有更完整字段，可作为后续增强项补充：

| 字段 | 含义 |
| --- | --- |
| `date` | 快照日期 |
| `sku_id` | SKU |
| `available_stock` | 可售库存 |
| `on_hand_stock` | 在手库存 |
| `in_transit_stock` | 在途库存 |
| `reserved_stock` | 已预留库存 |
| `stockout_flag` | 是否断货 |
| `warehouse_id` | 仓库编码 |

这类字段主要用于区分：

- 真正没有需求
- 还是因为没货导致卖不出去

## 7. 我们这边的使用原则

这轮补数或确认的目标，不是增加报表数量，而是让后续优化建立在更清楚的数据定义上。  
如果某些字段暂时无法提供，也请直接说明，我们会按保守口径处理。

保守口径包括：

- 对语义不清的负数订单不直接当作真实补货需求
- 对 `TYPE` 缺失行不强行归类
- 对暂时拿不到的生命周期字段先不做强假设
