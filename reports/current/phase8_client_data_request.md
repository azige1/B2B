# 预测模型数据需求与字段确认清单

## 1. 目的

本清单用于在进入下一个阶段之前，需要确认当前原始数据的业务语义，并补齐最有价值的新增数据。
优化目标不是重开模型路线，而是继续解决当前正式主线的剩余结构性短板：

- `>25 / blockbuster` 仍偏少补
- `repl0_fut0 / repl1_fut0` 弱信号高价值 SKU 识别不足
- 部分类目在 tail 场景下仍偏 under
- `true = 0` 但预测补货的小额错补仍然存在

> 我们在进入下一阶段优化前，想先把底层数据语义和少量关键补充数据确认清楚。当前最希望您协助提供或确认的内容有三类：  
>
> 1. 订单流水字段语义：`V_IRS_ORDERFTP` 中 `TYPE` 缺失、`QTY < 0`、以及重复行分别代表什么业务动作；  
> 2. 生命周期信息：建议提供按 `style_id` 为主的生命周期表，至少包含 `launch_date`、`planned_off_shelf_date`、`is_reorderable`；

## 3. 最值得补充的数据

## 4.1 生命周期表：建议正式提供

这是当前最值得新增的一张结构化表，建议直接新增一个表，或者在商品表上新增字段。

### 为什么需要

当前最难的样本是：

- 动态信号弱
- 但真实需求很高
- 尤其是高首单量、新品启动型 SKU

如果没有生命周期信息，模型很难区分：

- 新品启动期，应放量
- 稳定期，正常补货
- 尾货/下架期，应保守或抑制

### 建议表粒度

优先按 `style_id` 提供。  
如果确实存在同款不同 SKU 生命周期差异较大的情况，再补 `sku_id`。

### 建议字段

| 字段 | 含义 | 必需性 |
| --- | --- | --- |
| `style_id` | 款号主键 | 必需 |
| `sku_id` | 条码级 SKU 主键 | 可选 |
| `launch_date` | 正式上市日期 | 必需 |
| `first_orderable_date` | 买手首次可下单日期 | 推荐 |
| `first_delivery_date` | 首次实际发货日期 | 推荐 |
| `planned_off_shelf_date` | 计划下架日期 | 推荐有 |
| `actual_off_shelf_date` | 实际下架日期 | 可选 |
| `is_carry_over` | 是否延续款 | 推荐 |

### 最低可接受版本

如果暂时给不了完整生命周期表，最低也建议提供：

- `style_id`
- `launch_date`
- `planned_off_shelf_date`
